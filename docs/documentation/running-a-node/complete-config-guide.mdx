# Complete Configuration Guide & Platform Quirks

This guide contains the complete configuration files with enhanced comments and documents important Cosmos-specific behaviors that aren't immediately obvious to newcomers.

## Critical Platform-Specific Quirks

<Warning>
These are **non-obvious behaviors** unique to Cosmos SDK chains that can confuse even experienced operators from other blockchain platforms.
</Warning>

### 1. Peer Connection Errors Are Normal

**What you'll see:**
```log
ERR failed to dial peer err="dial tcp: i/o timeout"
ERR failed to connect to peer err="peer is on a different network"
ERR failed to add peer err="peer ID already exist"
ERR stopping peer for error err="EOF"
```

**This is completely normal!** Your node constantly tries to connect to peers from its address book. Many will fail because:
- Nodes go offline
- Firewalls block connections
- Nodes are on different networks
- Old addresses in the address book

**Only worry if:**
- You have 0 peers for extended periods
- Your node stops syncing
- You see the same error for ALL connection attempts

### 2. State Sync RPC Configuration Quirk

<Warning>
**Critical:** For state sync, you must specify 2 RPC servers, but using 2 different servers actually **harms** reliability. Always use the **same RPC server twice**.
</Warning>

**Correct configuration:**
```toml
[statesync]
enable = true
# Use the SAME server twice - this is intentional!
rpc_servers = "https://rpc.cosmos.network:443,https://rpc.cosmos.network:443"
```

**Why?** The state sync process requires consensus between RPCs. Using different RPCs can cause mismatches in light client verification. Using the same RPC twice satisfies the configuration requirement while ensuring consistency.

### 3. Pruning "Default" Isn't Actually Default

The `pruning = "default"` setting keeps the last 362,880 blocks (~25 days at 6s blocks). This uses significant disk space. For most non-archive nodes:

```toml
# Actually saves disk space
pruning = "custom"
pruning-keep-recent = "100"    # Keep last 100 blocks
pruning-interval = "10"         # Prune every 10 blocks
```

### 4. Gas Prices vs Gas Wanted

**Common confusion:** `minimum-gas-prices` in app.toml is NOT the same as `max-tx-gas-wanted` in the EVM section.

- `minimum-gas-prices`: Minimum fee per gas unit the node accepts
- `max-tx-gas-wanted`: Maximum total gas a single EVM transaction can request

### 5. The "Catching Up" False Positive

Your node might show `"catching_up": false` in the RPC status even when blocks are still syncing. Check the actual block height:

```bash
# Don't trust just catching_up status
curl -s localhost:26657/status | jq '.result.sync_info'

# Compare your height with the network
curl -s https://rpc.cosmos.network/status | jq '.result.sync_info.latest_block_height'
```

### 6. Address Book Corruption

If your node can't find peers, the address book might be corrupted:

```bash
# Nuclear option that often fixes peer issues
rm ~/.evmd/config/addrbook.json
# Node will rebuild it on restart
```

### 7. Genesis File Hash Mismatches

Even a single space difference in genesis.json causes sync failures. Always verify:

```bash
# Get the SHA256 hash
sha256sum ~/.evmd/config/genesis.json

# Compare with official hash
curl -s https://network.com/genesis.json | sha256sum
```

## State Sync Best Practices

### The Right Way

```toml
[statesync]
enable = true

# USE THE SAME RPC TWICE - This is intentional!
rpc_servers = "https://rpc1.network.com:443,https://rpc1.network.com:443"

# Get these from a block explorer or trusted source
trust_height = 15698000
trust_hash = "A1B2C3D4E5F6..."
trust_period = "168h0m0s"

# More fetchers = faster sync
chunk_fetchers = "10"
```

### Common State Sync Failures

1. **"Can't verify headers"**
   - Solution: Use the same RPC server twice
   
2. **"Snapshot timeout"**
   - Solution: Increase `chunk_request_timeout = "30s"`
   
3. **"No snapshots available"**
   - Solution: Try different RPC servers that actually serve snapshots

## Quick Diagnostic Commands

```bash
# Is my node actually syncing? (Don't trust catching_up alone!)
curl -s localhost:26657/status | jq '.result.sync_info.latest_block_height'

# How many peers do I really have?
curl -s localhost:26657/net_info | jq '.result.n_peers'

# Am I connected to the right network?
curl -s localhost:26657/status | jq '.result.node_info.network'

# Is my node stuck? Check if height is increasing
watch -n 2 'curl -s localhost:26657/status | jq .result.sync_info.latest_block_height'

# Nuclear reset if nothing works
evmd tendermint unsafe-reset-all
rm ~/.evmd/config/addrbook.json
```

## The "It's Not Working" Checklist

1. ✅ Genesis hash matches network?
2. ✅ Same software version as network?
3. ✅ System time synchronized?
4. ✅ Ports open in firewall?
5. ✅ Using correct chain-id?
6. ✅ Seeds/peers actually online?
7. ✅ Disk has enough space?
8. ✅ RAM not exhausted?

If all yes and still not working:
